How AI Agents & Claude Skills Work — Clearly Explained

▶ Watch on YouTube

Overview

A conversational deep-dive (with lots of whiteboard diagrams) on how to actually get good results out of modern AI coding agents. The core argument: the models are already exceptionally good, so quality now comes from context engineering, not from piling on configuration. The guest makes the case for keeping AGENTS.md/CLAUDE.md files minimal, leaning instead on skills — small instruction files loaded via progressive disclosure — and building those skills by walking the agent through a real workflow until a successful run can be codified. It closes on managing the context window, treating agents like new employees, and scaling up to sub-agents only once core workflows exist.

Key takeaways

  • Models are good now; context and the harness are what separate quality from slop.
  • ~95% of people don't need an AGENTS.md/CLAUDE.md — it's injected every turn and burns tokens. Reserve it for proprietary, always-relevant info.
  • Skills use progressive disclosure: only the name + description sit in context (e.g. 53 tokens) until the agent decides it needs the full body (e.g. 944 tokens).
  • Don't write or download skills blindly — walk the agent through the workflow step-by-step, get a successful run, then have it codify that run into a skill.
  • Recursively build skills: when the agent fails, ask for the error, fix it, then have it update the skill so the failure can't recur.
  • LLMs don't think — they predict tokens. Treat them like new employees, not all-knowing boxes.
  • Keep the context window lean (aim well under ~70–80%); models get "dumb" as it fills, and it costs more.
  • Code itself is context — a solid template or codebase removes the need to restate your stack.
  • Scale for productivity, not for show: start with one agent, add sub-agents only when a real workflow justifies them.
  • Downloaded third-party skills are an attack vector and usually lack the context of a successful run.

Key Concepts

ConceptDefinition
Skills (agent skills)Reusable instruction files (skill.md) holding a name, description, and detailed steps for a specific workflow. The agent loads only the name and description until it decides the skill is needed, then reads the full content.
Progressive disclosureA design pattern where only a skill's name and description are kept in context; the full body is loaded only when the agent determines it needs that skill, saving tokens.
Context windowThe bounded amount of information (system prompt, instructions, tools, codebase, conversation) a model can attend to at once, measured in tokens; performance degrades as it fills up.
Context (in agents)The assembled information a model needs to execute an action — system prompt, instruction files, skills, tools, codebase, and conversation history.
AGENTS.md / CLAUDE.mdProject-level instruction files injected into context on every turn. Useful only for proprietary or always-relevant guidance; otherwise they waste tokens that a skill could carry on demand.
System promptThe provider-supplied base instructions that guide how a model behaves — what to do and what not to do — present in every request.
Agent harnessThe software scaffolding around a model that manages context assembly, exposes tools, and lets the model call them; the model itself does not call tools directly.
Recursively building skillsAn iterative method: build a skill, run it, capture each failure, ask the agent for the error, fix it, then have the agent update the skill so the failure never recurs.
Building skills from successful runsRather than writing a skill upfront, walk the agent through the workflow step-by-step until a successful run exists, then have it codify that run into a skill so it has real context.
TokensThe units of text models process and bill on; instruction files cost tokens on every turn, while progressively disclosed skills cost only their name and description until used.
Context rot / degradationThe tendency of a model to produce worse output as its context window fills (e.g. past ~70-80%); keeping context lean improves both cost and performance.
LLMs as token predictorsLarge language models don't reason or understand; they map input to a vector space and predict the most probable next tokens, which only appears like thinking.
Sub-agentsSpecialized agents managed by a main agent, each with its own skills and context for a domain (marketing, business, personal). Valuable only after core workflows exist.
Scaling for productivity vs for coolThe principle of starting with one capable agent and adding skills/sub-agents only when a real workflow justifies it, rather than spinning up many agents and skills upfront.
Vibe codingBuilding software primarily by prompting AI agents to write the code, rather than hand-writing it — increasingly viable because models are strong at code generation.
Code as contextThe idea that an existing codebase or template itself supplies the agent the conventions and stack info it needs, removing the need to restate them in instruction files.
Templates renaissanceThe prediction that well-built starter templates regain value because a solid foundation becomes immediate, high-quality context for an agent to build upon.
CompactionWhen a conversation approaches the context limit, tools like Claude Code or Codex summarize/compress earlier context to free space.
Skills marketplace riskDownloadable third-party skills are both an attack vector (malicious instructions) and often low-quality, since they lack the context of a successful run for your workflow.
OpenClaudeAn open agent setup the speaker runs with its own email and scheduled checks (e.g. every 15 minutes) to triage sponsor emails and run multi-source report workflows.
Permanent underclassA speculative term for workers displaced once AGI arrives who lack AI/agent skills; the speakers push back on the 'permanent' framing as overstated.
TypeScript code generationModel providers have optimized agents to be especially strong at writing TypeScript, underpinning code-writing agent products.
